# elm-land Codegen (lando)

File-based routing and page dispatch code generation for the v3 admin SPA, inspired by [elm-land](https://elm.land/). The directory tree under `clients/admin/src/admin/pages/` is the single source of truth for routes: add a `.gleam` file, run `bin/dev`, and the router and page dispatch are regenerated automatically.

## What it generates

Two files from the same directory scan:

| Generated file | Package | Purpose |
|---|---|---|
| `shared/src/shared/admin/route.gleam` | shared | `Route` union type, `parse_route`, `route_to_path`, `href` |
| `clients/admin/src/generated/page_dispatch.gleam` | admin client | `PageModel`/`PageMsg` union types, `init`/`update`/`view`/`breadcrumb` dispatch |

Both files start with `// Generated by lando -- do not edit.` and are excluded from linting and code review.

## How it runs

`bin/dev` Phase 1.5 checks if any `.gleam` file under `pages/` is newer than the stamp file. If so, it runs:

```bash
cd lib/lando
gleam run -m lando
```

The tool scans the pages directory, generates both files, and prints the route count:

```
lando: generated route.gleam + page_dispatch.gleam with 103 routes
```

## Conventions

### File and directory naming

The directory tree maps directly to URL path segments. Files and directories use snake_case.

| Filesystem | URL segment | Route variant |
|---|---|---|
| `events.gleam` | `/events` (static) | `RegistrationEvents` |
| `id_.gleam` | `/:id` (dynamic Int) | `RegistrationEventsId(id: Int)` |
| `key_.gleam` | `/:key` (dynamic String) | `...Key(key: String)` |
| `new.gleam` | `/new` (static) | `RegistrationEventsNew` |

**Trailing underscore** marks a segment as dynamic. The parameter name is the stem without the underscore.

### Parameter type inference

Convention over configuration: names that are `id` or end in `_id` become `Int` parameters. Everything else becomes `String`. This covers the project's patterns without any annotation files.

### The home route

`home_.gleam` at the pages root is a special case. It produces a `Home` variant with no segments, matching `/en/admin` (the admin dashboard).

### Route variant names

Built by PascalCasing each segment in order:

```
pages/registration/orders/id_/payments/payment_id_/edit.gleam
  -> RegistrationOrdersIdPaymentsPaymentIdEdit(id: Int, payment_id: Int)
```

### Generated parse_route

Matches URL path segments with `[_lang, "admin", ...]` prefix. Static segments precede dynamic ones in match order so `/events/new` is tried before `/events/:id`. Int parameters are validated at parse time; a failed `int.parse` falls through to `NotFound(uri:)`.

```gleam
// Example generated arm:
[_lang, "admin", "registration", "events", id] ->
  case int.parse(id) {
    Ok(id_val) -> RegistrationEventsId(id: id_val)
    Error(_) -> NotFound(uri:)
  }
```

### Generated route_to_path

Reconstructs URL paths from route variants. Adjacent static segments are merged into a single string literal for efficiency:

```gleam
// Instead of: prefix <> "/registration" <> "/events" <> "/" <> int.to_string(id)
// Generates:  prefix <> "/registration/events/" <> int.to_string(id)
```

## Page contract

Every page module (whether a real implementation or a delegation wrapper) must export:

```gleam
pub type Model { ... }
pub type Msg { ... }

pub fn init(context: ClientContext, ..params) -> PageResult(Model, Msg)
pub fn update(context: ClientContext, model: Model, msg: Msg) -> PageResult(Model, Msg)
pub fn view(model: Model) -> Element(Msg)
pub fn breadcrumb(client_ctx: ClientContext, model: Model) -> List(breadcrumb.Crumb)
```

`PageResult` is a 4-tuple:

```gleam
pub type PageResult(model, msg) =
  #(model, Effect(msg), Option(#(FlashLevel, String)), Option(ClientContextMsg))
```

The `init` signature varies by route: parameterless routes get just `context`, routes with `Int` params get `context, id` (positional after context), and `String` params work the same way.

## Dispatch generation

`page_dispatch.gleam` produces:

- **PageModel** union: one variant per page wrapping that page's `Model` (e.g., `HomeModel(page_home.Model)`) plus `NotFoundModel`.
- **PageMsg** union: one variant per page wrapping that page's `Msg` (e.g., `HomeMsg(page_home.Msg)`).
- **init**: matches the `Route` variant, calls the corresponding page's `init` with extracted params, wraps the result via a `wrap()` helper.
- **update**: matches `(PageModel, PageMsg)` pairs and delegates.
- **view**: matches `PageModel`, calls page view, maps the element's Msg type.
- **breadcrumb**: matches `PageModel` and delegates.

Import aliases follow the pattern `page_` + snake_cased variant name:

```gleam
import admin/pages/settings/general as page_settings_general
import admin/pages/registration/events/id_ as page_registration_events_id
```

## Page types

There are three patterns used in the `pages/` directory:

**Real page modules** have their own `Model`, `Msg`, `init`, `update`, `view`, `breadcrumb` implementations. The dashboard (`home_.gleam`), settings pages, and reports are examples.

**List delegation pages** wrap an existing list module. The page file re-exports the inner module's types and delegates all four functions. This connects a URL to existing list/index logic without duplicating it.

```gleam
// pages/registration/discounts.gleam
import admin/discounts/list as inner

pub type Model = inner.Model
pub type Msg = inner.Msg

pub fn init(context context: ClientContext) -> PageResult(Model, Msg) {
  inner.init(context)
}
// ... update, view, breadcrumb delegate similarly
```

**Form/detail delegation pages**: `new.gleam` files that delegate to the sibling `id_.gleam` (or its inner module), passing a sentinel like `0` for the ID to signal "create mode."

## The lando package

Located at `lib/lando/`, compiled to Erlang. Three modules:

| Module | Role |
|---|---|
| `types.gleam` | `ScannedRoute`, `UrlSegment`, `ParamType` types |
| `scanner.gleam` | Recursively walks `pages/`, returns `List(ScannedRoute)` |
| `generator.gleam` | Produces both `route.gleam` and `page_dispatch.gleam` source text |

The scanner handles: recursive directory traversal, segment parsing, PascalCase variant name construction, and module path computation. The generator handles: route sorting (static before dynamic), pattern matching with Int validation, static segment merging, and dispatch code with effect/msg wrapping.

## Adding a new page

1. Create a `.gleam` file under `pages/` at the path matching your desired URL.
2. Implement the page contract (`Model`, `Msg`, `init`, `update`, `view`, `breadcrumb`), or delegate to an existing module.
3. Run `bin/dev`. The codegen detects the new file and regenerates both output files.
4. The route, parse_route pattern, URL builder, and page dispatch are all wired up automatically.

No manual route registration, no import lists to update, no dispatch cases to write.
